# 延时摄影自动化工具 + AI 照片筛选 — 设计文档

**日期**：2026-06-02
**平台**：macOS 桌面应用
**技术栈**：Electron（前端）+ Python/FastAPI（后端）

---

## 1. 项目概述

一个 macOS 桌面应用，包含两个独立功能：

1. **延时摄影自动流水线**：自动完成 BR → LRT → AE → PR 流程。用户只需在 LRTimelapse 中手动调整关键帧参数，其余全部自动完成。
2. **AI 照片筛选**：拍摄单张/人像时，从一批照片中自动挑选最佳时刻、最佳构图、最佳瞬间。

两个功能独立，互不依赖。

---

## 2. 流程定义（BR-LRT-AE-PR）

| 缩写 | 软件 | 作用 |
|------|------|------|
| **BR** | Adobe Bridge + Camera Raw | RAW 文件初始批处理 |
| **LRT** | LRTimelapse | 关键帧设置、Holy Grail 日转夜过渡、去闪烁 |
| **AE** | After Effects | 图像序列导入、防抖、渲染中间视频 |
| **PR** | Premiere Pro | 最终剪辑、增稳、调色、按规格导出 |

**自动化边界**：用户唯一的手动介入点是在 LRT 中调整关键帧的曝光/色彩参数。其余所有步骤自动完成。

## 4. 功能模块

### 4.1 延时流水线控制面板

> **重要架构决定（2026-06-02 二次修订）**：
> - LRTimelapse 6 没有任何命令行/脚本接口，全程图形界面操作，无法可靠自动化 → **LRT 阶段全程手动**。原「关键帧数量/去闪幅度/去闪遍数」参数由用户在 LRT 内手动设置，工具不提供这些控件。
> - **BR 阶段也改为手动**：用户的 Camera Raw 调整（透视矫正、镜头配置文件、消色差）每次拍摄都不同（镜头/焦段不同），无法用统一预设。因此工具不再「套预设」，而是帮用户「开门」——用 Adobe Bridge 打开该次延时文件夹、全选 RAW、进入 Camera Raw，用户手动调整后点"继续"。原「Camera Raw 预设 .xmp」字段移除。
> - 结论：流水线有 **两个手动暂停点（BR、LRT）** + 两个自动段（AE、PR）。
>
> **三次修订（2026-06-03）—— 去闪与增稳归位到 AE**：
> - **双重去闪**：用户在 LRT 手动去闪一遍后，AE 阶段再用 AE 2026 自带的原生 **Deflicker** 效果（`Plug-ins/Effects/Deflicker.plugin`）自动去闪第二遍。
> - **增稳在 AE 做**（不在 PR）：用 AE 自带的 **变形稳定器（Warp Stabilizer / matchName `ADBE SubspaceStabilizer`）**。PR 阶段不再增稳，只做导出。
> - 因此 AE 阶段 = 导入序列 → Deflicker 去闪 → 变形稳定器增稳 → 渲染 ProRes 4444；去闪与增稳参数都在表单可调。

| 参数 | 控件类型 | 说明 |
|------|----------|------|
| RAW 文件夹 | 路径选择器 | 输入素材 |
| 相机机型 | 下拉（可自定义添加） | 决定可选导出分辨率 |
| LRT 导出序列文件夹 | 路径选择器 | 用户在 LRT 里导出图像序列的目标文件夹，工具据此接管 AE |
| AE 去闪 | 开关 + 强度/时间半径 | AE 原生 Deflicker（LRT 之外的第二遍去闪） |
| AE 增稳 | 开关 + 结果 + 平滑度 + 方法 | AE 变形稳定器（结果：平滑运动/无运动；平滑度 %；方法：位置/位置缩放旋转/透视/子空间变形） |
| 分辨率 | 下拉（随机型动态填充） | 导出规格 |
| 帧率 | 数字输入 + 预设（1–120 fps） | 导出规格，可自由输入 |
| 导出格式 | 预设 / 手动 | H.264/H.265/ProRes（见 §4.4） |
| 输出路径 | 路径选择器 | 成片保存位置 |

**交互流程**：
1. 用户选择 RAW 文件夹并配置参数 → 点「开始处理」
2. **[BR 手动]** 工具用 Bridge 脚本打开 RAW 文件夹、全选、调出 Camera Raw → 用户手动调透视/镜头配置/色差 → 点"继续"
3. **[LRT 手动]** 工具打开 LRTimelapse 并显示操作清单 → 用户手动完成关键帧/第一遍去闪/自动过渡/导出图像序列到指定文件夹 → 点"继续"（工具校验序列文件夹已有图片）
4. **[AE 自动]** 工具用 ExtendScript 新建 AE 工程、导入序列、建合成 → 加 Deflicker（第二遍去闪）→ 加变形稳定器增稳 → 加入渲染队列 → `aerender` 渲染为 ProRes 4444 中间视频
5. **[PR 自动]** 工具用 ExtendScript 把中间视频导入 Premiere → 按导出格式导出成片（不增稳）

**各软件自动化能力（探测结论）**：

| 阶段 | 软件 | 方式 |
|------|------|-----------|
| BR | Adobe Bridge 2026 + Camera Raw | ⚙️ 半自动：脚本打开+全选+调出 ACR，用户手动调 |
| LRT | LRTimelapse 6 | ❌ 无 CLI，全程手动 |
| AE | After Effects 2026 | ✅ ExtendScript：导入序列 + 原生 Deflicker 去闪 + 变形稳定器增稳 + `aerender` 渲染 |
| PR | Premiere Pro 2026 | ✅ ExtendScript 导入 + 按格式导出（不增稳） |

### 4.2 相机机型与分辨率

机型选择器，选完后分辨率下拉自动填充该机型可选规格。预设示例：

| 相机 | 原生分辨率 | 可选导出分辨率 |
|------|-----------|--------------|
| Sony A7R IV | 9504 × 6336 | 原分辨率 / 8K / 4K / 1080p |
| Sony A7R V | 9728 × 6656 | 原分辨率 / 8K / 4K / 1080p |
| Sony A7 IV | 7008 × 4672 | 原分辨率 / 4K / 1080p |
| Canon R5 | 8192 × 5464 | 原分辨率 / 8K / 4K / 1080p |
| Nikon Z9 | 8256 × 5504 | 原分辨率 / 8K / 4K / 1080p |
| 富士 GFX100S | 11648 × 8736 | 原分辨率 / 8K / 4K / 1080p |

**用户可自定义添加机型**（输入机型名 + 原生分辨率），配置持久化到 `cameras.json`。

通用分辨率规格：
- 8K：7680 × 4320
- 4K：3840 × 2160
- 2K：2048 × 1080
- 1080p：1920 × 1080
- 原分辨率：跟随机型

### 4.3 照片筛选面板

输入：支持两种来源
- A) 一次连拍序列（如 100 张里挑最好的几张）
- B) 一整天/一次外拍的所有照片（跨场景混合）

| AI 评分维度 | 技术实现 |
|------------|---------|
| 清晰度 | Laplacian 方差（OpenCV） |
| 人脸质量 | MediaPipe 检测 + 眼睛开合/表情评分 |
| 构图 | 三分法、主体居中/偏移检测 |
| 综合美学 | NIMA 美学评分模型 |

**结果展示**：网格缩略图，评分最高的照片打金色标记。用户可一键复制/移动最佳照片到新文件夹。

## 5. 数据流（延时任务）

```
RAW 文件夹
    │
    ▼ [BR 阶段 - 手动] Python 调 Bridge ExtendScript 打开文件夹+全选+调出 ACR
    │  用户手动：透视矫正 → 镜头配置文件 → 消色差 → 点"继续"
    │
    ▼ [LRT 阶段 - 手动] Python 打开 LRTimelapse + 显示操作清单
    │  用户手动：关键帧 → 第一遍去闪 → 自动过渡 → 导出图像序列到指定文件夹
    │  用户点"继续"（工具校验序列文件夹已有图片）
    │
    ▼ [AE 阶段 - 自动] Python 生成 ExtendScript 建工程：
    │  导入序列 → Deflicker 第二遍去闪 → 变形稳定器增稳 → 入渲染队列
    │  → aerender 命令行渲染：→ ProRes 4444 中间视频
    │
    ▼ [PR 阶段 - 自动] Python 生成 Premiere ExtendScript
    │  导入中间视频 → 按导出格式导出（不增稳）
    │
    最终 MP4/MOV 文件
```

## 7. 错误处理

| 失败场景 | 处理策略 |
|---------|---------|
| 软件未安装（LRT/AE/PR） | 启动前预检测，缺失则明确提示并阻止开始 |
| 某阶段崩溃 | 保留中间产物，标记失败阶段，支持**断点续跑** |
| 用户调参阶段超时 | 不超时，无限等待用户点"继续" |
| RAW 文件损坏/格式不支持 | BR 阶段校验，列出问题文件，让用户选择跳过或中止 |
| 渲染磁盘空间不足 | 开始前估算所需空间并检查 |
| AI 模型加载失败 | 照片筛选降级——仍可用基础清晰度评分，跳过美学模型 |

**进度反馈**：每个阶段通过 WebSocket 实时推送进度（百分比 + 当前操作日志）到前端看板。

## 8. 测试策略

- **AI 评分模块**：单元测试，准备已知质量的测试图片集（清晰/模糊、睁眼/闭眼、好/差构图），验证评分排序符合预期。
- **流水线各阶段**：mock 外部软件调用，验证参数正确传递、XMP 写入正确、断点续跑逻辑。
- **集成测试**：用一小组真实 RAW 文件跑完整流水线，人工验证成片。
- **前后端通信**：测试 WebSocket 进度推送、错误传播。

## 9. 范围边界（YAGNI）

- 仅 macOS，不做 Windows
- 不做拍摄前的气象/光线预测（用户明确表示与延时无关）
- 不做云端/协作功能，纯本地工具
- 照片筛选只做"挑选+标记+复制移动"，不做修图

# Skyfire 后端预测闭环(Plan C-2)设计

日期:2026-07-05。前置:Plan A/B/C-1/D/D-2/D-3 已完成(CLI 管道、AWS 云图、可判读渲染、案例卡/经验笔记)。
领域依据:`docs/superpowers/knowledge/2026-07-05-firecloud-domain-knowledge.md`(§3.2 预报当前瞻底子+卫星实测为准、§5 读图配方、§6 量化框架)。

## 0. 目标(用户原话口径)

**终极目标 = 准确预测晚霞/朝霞。** 每天多次根据 EC/GFS/ICON 等模式更新预测;临近日出日落自动结合实时卫星云图判断;预测以**百分数**形式给出**质量%**(烧多好)与**概率%**(烧不烧);全程后台自动获取数据完成更新;用户反馈(照片/实际得分/"预报不准")触发后台复盘学习,改进后续预测。

本 spec 只覆盖**后端闭环(CLI/调度优先)**。FastAPI 服务化与 Taro/微信前端是下一个子项目,不在此内。

## 1. 分层架构:免费层高频跑,LLM 只在检查点烧

```
免费层(不烧 Anthropic key,tick 每 30min 可全量刷):
  Open-Meteo 多模式点预报/网格云图 → 规则分+一致性
  AWS HSD 实时卫星 → 渲染判读图 + box_cloudiness 实测云量 + 趋势外推(drift)
LLM 层(烧 key,只在检查点/门控触发):
  Claude 多模态:判读图 + 趋势 + RAG 相似案例与教训笔记 → 质量% / 概率% / 理由 / 置信
```

关键原则(实证依据 2026-06-26):**预报云量当前瞻底子看趋势,不当精确真值;燃烧时刻云量以卫星实测/外推为准**。LLM 输入必须同时含预报层与实况层,并明示两者分歧。

## 2. 检查点调度(挂在现有 tick / launchd 30min)

每天每城每天象(晚霞为例,朝霞镜像):

| 检查点 | 时机 | 输入 | LLM |
|---|---|---|---|
| C1 早间展望 | ~11:00(模式更新后) | 多模式预报 + 预报云图网格 | ✓ |
| C2 临近-2h | T−2h | 预报刷新 + 实况卫星连帧 + 趋势外推 | ✓ |
| C3 终版 | T−40min | 实况为主,外推到燃烧时刻 | ✓ |
| 门控加跑 | 检查点之间每 30min | 免费层重算;仅当概率/质量摆动 > **15 个百分点**才唤 LLM,否则只用免费层修正数字、沿用上次解读 | 条件✓ |

- tick 每次运行:先免费层刷新 → 判断"是否到达检查点或触发门控" → 是则跑 LLM 检查点并落库/推送。
- 幂等:同一检查点同日只跑一次(数据库判重,类似 notifications 表)。
- 无 key:检查点照常落**免费层版本**(规则+一致性+外推得出的百分数),LLM 字段标 `pending`。

## 3. 预测输出(百分数,版本化落库)

新表 `predictions`:

```
id, date, city, event, checkpoint(text: c1|c2|c3|gated|manual),
probability_pct REAL, quality_pct REAL,      -- 0-100
confidence TEXT,                              -- high/medium/low/degraded
rule_score REAL, sat_cloud_pct REAL, trend TEXT,  -- 免费层快照(溯源)
llm_status TEXT,                              -- done|pending|skipped
reasoning TEXT, risks TEXT,                   -- LLM 理由(无则空)
created_at
```

- 查询/推送永远取该日该天象**最新一版**;历史各版保留 → 事后可复盘"预测如何随时间收敛/翻车"(本身是学习材料)。
- 百分数换算基线(无 LLM 时):`概率% = f(规则分, 一致性, 实测云量在甜区与否)`、`质量% = 规则分×10 经外推修正`——公式初版简单可解释,由回测迭代;LLM 在场时以 LLM 输出为准(免费层数字仅作为它的输入之一,LLM 可高于或低于它——例如卫星实测与预报打架时)。

## 4. LLM 层(可插拔、分层、缓存、降级)

- **模型分层**:日常检查点 `claude-haiku-4-5`;"疑难"升级 `claude-sonnet-5`(触发:C1 与 C3 分歧>25 个百分点、或复盘发现预测vs实际背离≥4 分、或 RAG 无相似案例)。
- **prompt caching**:system(读图配方§5+量化口径)与 RAG 模板走缓存;图像逐次新传。
- **调用形态**:输入 = 判读图(IR 连帧 3-4 张 + 网格预报图 1 张)+ 免费层数字 + 相似案例及教训笔记;输出 JSON `{probability_pct, quality_pct, reasoning, risks, confidence}`。
- **降级**:无 key/失败 → 免费层版本落库,`llm_status=pending`;不阻塞任何链路。
- **catchup 命令**:`skyfire catchup`——有 key 时批量补跑 pending(预测版与复盘均可补),按时间序补齐笔记。
- 成本预期(单城,晚霞+朝霞):约 6-8 次 Haiku 调用/日 ≈ 每月 ¥20-40;疑难 Sonnet 少量;缓存后更低。

## 5. 反馈闭环(学习链路)

- **`skyfire feedback --date --event [--score 0-10] [--photo 路径] [--wrong]`**:
  1. 落 `actual_score` / photos 表(照片拷贝入 data/photos/,路径入库);`--wrong` 等价"预报不准"信号(不给分也触发复盘)。
  2. 闭合案例后**自动触发后台复盘**:回填当天帧(若缺)→ Claude 读"当天判读图 + 用户照片 + 各版预测记录" → 按五段口径解释"为什么准/不准、哪个因子骗了预测" → 写 `case_notes(author='llm')`;无 key → pending。
- 复盘重点看**预测轨迹 vs 实际**:C1/C2/C3 各版谁对谁错(如"C1 信了预报 100% 云量,C3 卫星实测 25% 已纠偏但没纠够")——教训直接可操作。
- 笔记入 RAG(已有 Task 11 通路),下次预测检索到相似案例时连教训一起给 LLM。

## 6. 趋势外推强化(nowcast → 预测输入)

- 现有 `estimate_shift`/`extrapolated_corridor` 保留;新增**观测点云量外推**:用位移向量把当前实测云场平移到燃烧时刻,估计届时观测点上空云量%与走廊状态,作为 C2/C3 的核心输入。
- 朝霞夜间厚度盲区:可见光在燃烧前不可用(物理限制);近期以红外连帧趋势为主;**多波段夜间微物理合成(测厚)列为后续增强**,本期不做。

## 7. 错误处理与降级(沿 spec 8 惯例)

- 数据源单点失败不崩链路:预报缺 → 用上一版+标 degraded;卫星缺 → 纯预报层;LLM 失败 → pending。
- tick 内单城/单天象失败互不影响;所有网络层失败静默记录、下个 tick 重试。

## 8. 测试策略

- 免费层与调度:纯函数+MockTransport 全覆盖(检查点判定、门控阈值、幂等、百分数换算)。
- LLM 层:假客户端(沿 test_llm 模式)测输出解析/降级/pending/catchup。
- 端到端:真跑一天的 tick 序列(可用历史日期模拟时钟)出 C1→C3 预测轨迹;feedback 触发复盘写入笔记。

## 9. 非目标(本期不做)

- FastAPI/HTTP API、Taro/微信小程序、多用户与鉴权(下一个子项目)。
- 多波段夜间测厚合成、SLIDER 彩图。
- 机器学习训练;云海(cloud_sea)链路沿用现状不动。

## 10. 里程碑验收

1. tick 全自动跑出每日 C1/C2/C3 百分数预测轨迹并推送(现有 Bark/Server酱)。
2. `feedback` 一条命令闭合案例并产出(或 pending)复盘笔记。
3. 无 key 全链路不断;配 key 后 `catchup` 补齐。
4. 回测新增"概率%/质量% vs 实际得分"的相关性与命中率报表(替代已证伪的纯规则 ρ)。

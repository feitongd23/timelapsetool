# 火烧云 / 云海预测助手 — 设计文档

日期:2026-07-03
状态:待用户评审
代号:skyfire(暂定,后续可改)

## 1. 背景与目标

用户是城市风光摄影师,常拍朝霞/晚霞火烧云与城市云海,目前靠手动查看 Windy
(ECMWF/GFS 等模式)做前期判断。目标是把这个判断过程产品化:

- **自动预测**:对指定城市输出火烧云指数(朝霞/晚霞)与云海指数(按机位),含置信度与人话解读
- **经验积累**:每次天象事件自动归档"预报数据 + 卫星云图 + 实际打分/实拍图",让预测随使用越来越准
- **产品形态**:微信小程序 + App(Taro 一套代码双端),先自用+小圈子(体验版/TestFlight),架构预留公开运营

第一验证城市:北京。架构城市无关——任意经纬度 + 机位配置即可接入新城市。

## 2. 非目标(MVP 明确不做)

- 不批量爬取小红书/墨迹天气等第三方平台(无公开 API,合规风险;冷启动学习资料由用户一次性人工整理,之后靠用户打分+传图)
- 不做正式上架与公开运营(ICP 备案、气象发布合规、微信/苹果审核留到公开化阶段)
- 不做全国城市批量开通,只保证"加一个城市 = 加一份配置"
- 不训练/微调模型——"学习"是检索式的(相似案例 RAG + Claude 多模态解读)

## 3. 架构总览

```
┌─ 数据采集(Mac 定时任务)──────────────────┐
│  Open-Meteo 多模式预报(EC/GFS/ICON/CMA)   │
│  Himawari-9 卫星真彩/红外图(10 分钟/张)    │
│  日出日落/晨昏蒙影时刻(本地天文计算)        │
└──────────────┬────────────────────────┘
               ▼
┌─ 预测引擎(Python / FastAPI)─────────────┐
│  规则层:火烧云指数 + 云海指数(按机位)      │
│  一致性层:多模式分歧 → 置信度              │
│  临近修正:卫星实况外推 → 评分调整(±2)      │
│  经验层:相似案例检索 → Claude 解读+修正     │
└──────────────┬────────────────────────┘
               ▼
┌─ 经验库(SQLite + 图片文件目录)────────────┐
│  案例 = 预报快照 + 卫星图序列 + 打分 + 实拍图 │
└──────────────┬────────────────────────┘
               ▼
┌─ 客户端(Taro,一套代码双端)───────────────┐
│  微信小程序(体验版)+ iOS App(TestFlight)  │
│  今日预报 / 临近推送 / 打分传图 / 案例库      │
└───────────────────────────────────────┘
```

部署:后端跑在用户 Mac 上(launchd 定时任务 + FastAPI 常驻),经 Cloudflare
Tunnel 暴露 HTTPS 供小程序体验版访问;数据层为单文件 SQLite + 本地图片目录,
将来平移到云服务器只需拷贝数据目录,架构不变。

## 4. 数据源

| 数据 | 来源 | 说明 |
|---|---|---|
| 多模式点预报 | Open-Meteo Forecast API(免费) | 同一请求返回 ECMWF/GFS/ICON/CMA 等模式的分层云量(高/中/低)、湿度、能见度、风;与 Windy 展示的数据同源 |
| 气溶胶 | Open-Meteo Air Quality API(免费) | AOD/PM,用于霾修正 |
| 通道剖面 | 同上,点阵采样 | 沿日落/日出方位角方向取采样点(约每 25 km 一点,至 400 km),重建光通道上的逐小时云量剖面 |
| 卫星实况 | Himawari-9(公开) | 10 分钟/张,白天真彩、夜间红外(朝霞预测用红外);约 1-2 km 分辨率 |
| 历史卫星图 | Himawari-8/9 AWS 开放数据归档 | 2015 年至今全量,供冷启动回填 |
| 历史气象 | Open-Meteo Historical / Previous Runs API(免费) | ERA5 再分析 + 2022 年起的历史预报存档,回填时优先用"当天预报当时怎么说"的存档 |
| 日出日落 | 本地计算(astral 等库) | 含民用晨昏蒙影时刻,确定"烧云窗口"和卫星取帧窗口 |

Windy 自身 API 不接入:其数据与 Open-Meteo 同源(均为 ECMWF/GFS 等模式),
且 Windy API 收费、条款限制转发。"未来时刻的云图"由模式数据点阵采样重建
(MVP),完整格点场渲染(ECMWF Open Data / NOAA GFS GRIB)留作二期。

## 5. 预测引擎

### 5.1 规则层 — 火烧云指数(0-10,朝霞/晚霞各一次)

因子(初版权重为经验值,以回测校准为准):

1. **画布云**:日落/日出时刻本地上空高云+中云覆盖率,40-70% 区间最佳;
   卷云(高云)权重高于中云
2. **通道透光**:方位角方向 100-400 km 采样点的低云与总云量;通道堵塞为
   一票否决级重罚(最常见的"预报有云但没烧"原因)
3. **云底高度/结构**:越高越易烧,被照亮窗口越长
4. **气溶胶/能见度**:AOD 过高(霾/沙尘)按梯度扣分
5. **降水/对流**:窗口期本地降水直接压分

### 5.2 规则层 — 云海指数(0-10,按机位)

因子:夜间辐射降温条件(晴、风 < 3 m/s、昼夜温差)、近地层湿度饱和度、
低云/雾顶高度与机位海拔差(雾顶须低于机位)、日出前后大气稳定度。

机位为配置数据(名称/经纬度/海拔),北京初始机位:雾灵山、百花山、妙峰山,
用户可自行增删。

### 5.3 一致性层 — 置信度

对每个关键因子比较各模式(EC/GFS/ICON/CMA)的预报值:分歧小 → 置信度高;
分歧大 → 明确输出"模式打架,建议赌一把/放弃"。置信度随预报时效临近更新
(提前 72h 起报,逐 6h 刷新)。

### 5.4 临近修正层 — 卫星外推

窗口前 2 小时拉取最近数帧 Himawari-9 实况,估计云区移动矢量,外推判断:
通道是否真干净、画布云是否到位。对基础分做 ±2 以内修正,触发推送
("最后反悔机会")。

### 5.5 经验层 — 相似案例检索 + Claude 解读

- 检索:以当天预报的因子向量(画布云量、通道剖面、湿度、AOD、季节等)在
  经验库中检索 top-3 相似案例
- 解读:将当天数据 + 相似案例(含卫星图,Claude 多模态直接看图)交给
  Claude,输出:修正评分建议、一段人话解读(引用相似案例的实际结果)、
  风险点提示
- Claude 不做基础算术,只做案例类比与多因素权衡;规则分与 Claude 修正分
  同时存档,便于事后对比谁更准

## 6. 经验库

SQLite 单文件 + 本地图片目录。表结构:

- `cases`:id、日期、城市、天象类型(朝霞/晚霞/云海)、事前规则分、事前
  Claude 修正分、实际打分、数据来源(冷启动/用户反馈/自动归档)
- `forecast_snapshots`:case_id、模式名、快照 JSON(分层云量、湿度、AOD、
  通道采样点序列)、起报时间
- `satellite_frames`:case_id、时间戳、通道(真彩/红外)、文件路径
  (窗口前后 2 小时,每 20 分钟一帧,约 12 帧/案例)
- `photos`:case_id、user_id、打分、图片路径、备注
- `spots`:城市、机位名、经纬度、海拔(云海用)
- `users`:小圈子用户(微信 openid),打分/传图归属

**自动归档**:每天不论好坏,朝霞/晚霞/云海窗口的预报快照与卫星帧全部落库
(自动积累负样本);用户打分到达后案例即"闭环"。

### 6.1 冷启动(一次性学习资料导入)

- 输入:用户整理的清单(CSV 或文件夹):日期、城市、天象类型、实际打分,
  配图可选(来源不限,仅入本地库不对外展示;带 EXIF 的照片可自动提取时间
  地点)
- 回填:逐条自动抓取当天历史预报存档(优先)/ERA5 + Himawari 归档帧,
  重建完整案例
- 建议规模:正样本 30-50 条 + 负样本 20-30 条(重点收录"预报看着行但没烧"
  的假阳性日,对案例类比教学价值最高)
- 此后不再依赖人工整理,增量全部来自用户打分+传图与每日自动归档

## 7. 客户端(Taro 双端,MVP 四页)

1. **今日**:朝霞/晚霞/云海三卡片——指数、置信度、Claude 一句话解读、
   相似历史案例缩略图;下拉城市切换(MVP 仅北京)
2. **临近**:窗口前 2 小时的卫星实况图 + 修正结论;推送经微信订阅消息
   (App 端用本地通知)
3. **反馈**:窗口结束后推送提醒打分(0-10)+可选传图;含"10 秒补录"入口
   (给过去某天补一条打分,系统自动回填该日数据成案例)
4. **案例库**:按时间/评分翻历史案例,预报 vs 卫星实况 vs 实拍图三方对照

## 8. 错误处理与降级

- Open-Meteo 超时/失败:重试后降级为单模式(缓存的上一轮 EC),标注
  "数据不全,置信度降级"
- Himawari 发布延迟/缺帧:跳过临近修正,输出规则分并标注"缺卫星修正"
- Claude API 失败:只出规则分,解读栏显示"AI 解读暂缺",不阻塞推送
- Mac 休眠错过定时任务:launchd 唤醒后补跑;所有任务幂等(按"城市+日期+
  窗口"去重)
- Tunnel 断连:客户端显示最后成功更新时间,超过阈值提示"数据可能过期"

## 9. 测试与验收

- **回测(首要验收)**:冷启动导入后,规则模型对全部历史案例出分,与实际
  打分算 Spearman 相关性;以此校准因子权重与阈值,校准前后对比留档
- **单元测试**:数据采集层 mock API 响应;评分函数用构造的极端场景
  (通道全堵/全晴无云/模式完全分歧)验证边界行为
- **端到端**:"模拟一天"脚本——注入某历史日数据,走完 采集→评分→Claude
  解读→推送→打分回填 全链路
- **持续验证**:上线后每条案例自动记录"规则分 vs Claude 修正分 vs 实际分",
  月度看准确率趋势

## 10. 风险与合规

- **阈值未经北京校准**:初版规则分预期"明显好于瞎猜、不及成熟服务",靠
  回测+反馈闭环迭代(sunsetbot 亦是多年调优)
- **第三方平台内容**:不做批量爬取;冷启动图片仅本地留存不对外展示
- **公开化前置项**(路线图,MVP 不做):国内云服务器+ICP 备案、微信/苹果
  审核、气象预报发布合规(以"拍摄指数"形态规避直接发布天气预报)、
  Himawari/Open-Meteo 商用条款复核
- **成本**:MVP 阶段仅 Claude API 调用(自用频率约 $5-15/月),数据源全免费

## 11. 路线图(MVP 之后)

1. 云海机位扩展与机位级历史统计
2. 格点云图渲染(ECMWF Open Data / GFS GRIB → 小程序地图动画)
3. 多城市开通与用户自助添加城市
4. 公开运营:云服务器迁移、备案、上架、众包打分体系
